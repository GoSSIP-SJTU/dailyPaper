今天为大家推荐的是来自ICSE 2021的一篇论文——“If It’s Not Secure, It Should Not Compile: Preventing DOM-Based XSS in Large-Scale Web Development with API Hardening”。

之前的大多数工作都是在软件生命周期的后期处理XSS问题，例如对整个应用进行静态分析，自动化测试以及运行时保护等。随着软件系统越来越复杂，安全漏洞引起的损失和修复所需开销急剧增加，而这些方法并不能够很好的消除复杂应用中的XSS漏洞。本文中，谷歌的安全研究人员分享了他们提出的软件工程范式API hardening，在软件开发的早期阶段，通过一系列技术控制，将XSS漏洞扼杀在摇篮里。


API hardening主要有两个部分：

  • 嵌入在编译器中的静态分析器，禁止使用有可能造成XSS的APIs。

  • 设计一系列安全类型和APIs来替代XSS-prone APIs。


首先需要识别危险的DOM APIs。作者根据Web IDL（接口定义语言）手动分析和搜集XSS sink点。总共确认了五种类型的XSS sinks：代码执行sinks，URL导航sinks，URL加载sinks，HTML sinks和CSS sinks。在编译时，安全分析器遍历AST 对XSS-prone APIs给出错误提示。


由于在开发中，危险APIs被识别和禁用，开发人员需要一套替代方案。所以作者设计了一系列安全APIs，限制DOM和JS引擎的交互，从而使得开发人员即使不了解每个组件的安全特性，也不会写出有漏洞的代码。安全APIs的设计由以下三个方面构成：

  • Safe types。每个安全类型对应上述的sink点类型。安全类型的值一旦创建就不可改变，并且禁止任何代码访问其内部属性。本质上是对原始字符串wrapper。


  • Safe builders。安全类型变量的值只能从Safe builders的APIs中产生，包括从字面量，sanitizers，模板以及手动审查过的源中构建。

  • Safe sinks。Safe sinks是原始APIs的wrappers，只接受类型为Safe type的值。



在设计完这一套开发范式后，需要考虑其实际部署。


对于大型的项目来说，用新的hardened APIs全部重构代码是不现实的。谷歌对于历史上XSS风险较高的代码进行重构，而其他遗留代码则等待产品的迭代而逐渐的移除。


对于数万名Web开发者来说，他们需要适应新的安全规范。为了很好的帮助开发者，编译器会显示错误信息和修补方式，API Hardening团队也会提供很好的文档和咨询。在开发者无法只使用安全APIs来完成某项功能时（极少数情况下），可以提出对某个文件的豁免。


在2018年，谷歌开始对JS代码进行编译时审查，并且持续的监控API Hardening的使用情况。作者对这几年谷歌生成环境中的API Hardening实践情况进行了分析。

可以看到，在应用API Hardening后DOM-Based XSS漏洞数量明显减少，遗留代码中违反安全的APIs也逐渐减少。结果表明了谷歌推出的这一套软件开发范式是很有价值的，有效的防止了XSS漏洞被引入到产品当中。
